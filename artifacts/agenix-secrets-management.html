01

What agenix does

Mental Model

agenix is a thin wrapper that lets you commit encrypted secret files into your Nix repo. Each .age file is a secret encrypted to one or more public keys. At activation time, the host decrypts its own secrets using the private key it already has — its SSH host key.

The core problem: NixOS configurations live in git. Plaintext secrets (SSH private keys, API tokens, passwords) cannot safely go in git, even a private repo. You need a way to commit the ciphertext and decrypt it on the target machine at deploy time.

agenix's answer: encrypt each secret to the public portion of each host's SSH host key (/etc/ssh/ssh_host_ed25519_key.pub). At activation, agenix decrypts the secret using /etc/ssh/ssh_host_ed25519_key and writes it to /run/agenix/<name> — a RAM-backed tmpfs that never hits disk.

You can also encrypt to your own user key (age key or SSH key), which lets you decrypt secrets on the developer machine when editing or re-keying. Host keys are for deployment; user keys are for authoring.

agenix is good for
  • SSH private keys per-user or per-host
  • API tokens and bearer credentials
  • Wireguard private keys
  • TLS private keys
  • Database passwords
agenix is not for
  • Secrets that must exist before SSH host keys are generated
  • Structured config (use sops-nix for YAML/JSON bundles)
  • Secrets needing runtime rotation without redeploy
  • Very large binary blobs
02

age encryption primer

age (Actually Good Encryption) is a file encryption tool. It encrypts a payload to one or more recipients, where each recipient is identified by a public key. Any matching private key can decrypt the result. This is the same asymmetric pattern as GPG but with a smaller, audited design.

age supports two key types as recipients:

Key type Public key format Typical use in agenix
age native key age1… Author / developer key for local editing
SSH ed25519 key ssh-ed25519 AAAA… Host SSH host key — what machines use to decrypt at activation
SSH RSA key ssh-rsa AAAA… Supported but prefer ed25519 for new setups

A single .age file can be encrypted to many recipients simultaneously. The same ciphertext is recoverable by any of the listed private keys. This is how you grant access to multiple hosts: encrypt once to all their public keys, commit the single file.

ℹ
age converts an SSH ed25519 public key to an age public key internally — you don't need to maintain a separate key file. You can paste the SSH host public key directly into the agenix manifest and it works.
03

The secrets manifest

agenix uses a Nix file — conventionally secrets/secrets.nix — as the manifest. It is not a NixOS module; it is a standalone Nix expression evaluated only by the agenix CLI tool. It maps each .age filename to the list of public keys that can decrypt it.

secrets/secrets.nix nix 
let
  # Host SSH host keys — pulled from each machine's
  # /etc/ssh/ssh_host_ed25519_key.pub after first boot.
  jaro    = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... jaro host key";
  everest = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... everest host key";
  fuji    = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... fuji host key";

  # Author / developer key — lets you re-encrypt locally without
  # logging into a machine. Generate with: age-keygen > ~/.age/key.txt
  brojo   = "age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p";

  # Convenience groups
  allHosts   = [ jaro everest fuji ];
  desktops   = [ jaro everest ];
  allKeys    = [ brojo ] ++ allHosts;

in
{
  # brojo's SSH identity — deployed to all three machines.
  # Any of their host keys (or the author key) can decrypt it.
  "brojo-ssh-key.age".publicKeys = allKeys;

  # A secret only the headless server needs — fuji + author only.
  "fuji-wireguard-key.age".publicKeys = [ brojo fuji ];

  # A secret that should only exist on desktop machines.
  "github-token.age".publicKeys = [ brojo ] ++ desktops;
}
◆
The manifest only tells the agenix CLI who can decrypt. The NixOS activation script independently knows which .age file to decrypt (you tell it in your module config). The manifest doesn't need to know about NixOS modules at all.

The actual .age files live alongside the manifest in secrets/. They are binary ciphertext files, safe to commit to a public repo. The directory looks like:

directory layout sh 
secrets/
  secrets.nix              # manifest — which keys decrypt which files
  brojo-ssh-key.age        # encrypted private key
  fuji-wireguard-key.age   # only fuji can decrypt at activation
  github-token.age         # only desktops can decrypt
04

Flake integration

Add agenix as a flake input, then pass its NixOS module to every host that needs secrets. The agenix overlay or package gives you the CLI tool in your dev shell.

flake.nix — inputs block nix 
inputs = {
  nixpkgs.url      = "github:nixos/nixpkgs?ref=nixos-unstable";
  home-manager = {
    url = "github:nix-community/home-manager";
    inputs.nixpkgs.follows = "nixpkgs";
  };
  # Add agenix
  agenix = {
    url = "github:ryantm/agenix";
    inputs.nixpkgs.follows = "nixpkgs";
  };
};
flake.nix — outputs / nixosConfigurations nix 
outputs = { self, nixpkgs, home-manager, agenix, ... }@inputs: {
  nixosConfigurations.jaro = nixpkgs.lib.nixosSystem {
    system = "x86_64-linux";
    specialArgs = { inherit (inputs) home-manager agenix; };
    modules = [
      ./lib/hardware/machines/brojo-thinkpad-p14s-gen2
      (import ./lib/hosts/desktop.nix { hostname = "jaro"; })

      # Add the agenix NixOS module to this host
      agenix.nixosModules.default

      home-manager.nixosModules.home-manager
      {
        home-manager.useGlobalPkgs    = true;
        home-manager.useUserPackages  = true;
        home-manager.users.brojo       = import ./lib/users/brojo/home.nix;
        # Expose agenix to home-manager modules via extraSpecialArgs
        home-manager.extraSpecialArgs  = { inherit agenix; };
      }
    ];
  };
};

Add the agenix CLI to your dev shell so you can create and edit secrets:

flake.nix — devShell nix 
default = pkgs.mkShell {
  buildInputs = with pkgs; [
    git
    nix
    nixos-rebuild
    agenix.packages.x86_64-linux.default  # gives you the agenix CLI
  ];
};
05

Using secrets in NixOS modules

Once agenix.nixosModules.default is in a host's module list, the age.secrets option becomes available. Each entry tells agenix where to find the .age file and what owner/permissions to apply. The decrypted content lands at the path in age.secrets.<name>.path.

lib/hosts/server.nix — declaring a secret nix 
{ config, ... }:
{
  age.secrets.wireguard-private-key = {
    file  = ../../secrets/fuji-wireguard-key.age;
    owner = "root";
    mode  = "0400";
  };

  networking.wireguard.interfaces.wg0 = {
    privateKeyFile = config.age.secrets.wireguard-private-key.path;
    # path resolves to /run/agenix/wireguard-private-key at runtime
  };
}
⚠
Never hardcode the path as /run/agenix/…. Always reference config.age.secrets.<name>.path — the runtime path is an implementation detail that can change.
Option Default Notes
file required Path to the .age file in the repo
owner "root" Unix user that owns the decrypted file
group "root" Unix group
mode "0400" Octal permissions
path read-only The runtime path; reference this in other options
symlink true Whether to symlink into /run/agenix/
06

Using secrets with Home Manager

Important distinction

Home Manager does not have a built-in agenix integration. Secrets are declared at the NixOS level (in a NixOS module), then referenced from home-manager via config passthrough or home.file. Home Manager runs as the user; agenix decryption runs as root during system activation.

The cleanest pattern for deploying a user SSH key is to declare the secret in a NixOS module with the correct owner, then have Home Manager symlink or reference the decrypted path. Two approaches:

Approach A — NixOS module owns the secret, home.nix symlinks it nix 
# In a NixOS module (e.g. lib/users/brojo/user.nix or lib/hosts/desktop.nix)
age.secrets.brojo-ssh-key = {
  file  = ../../secrets/brojo-ssh-key.age;
  owner = "brojo";
  mode  = "0600";
};

# In lib/users/brojo/home.nix — symlink from ~/.ssh/id_ed25519
home.file.".ssh/id_ed25519".source =
  config.age.secrets.brojo-ssh-key.path;
ℹ
Passing config to home.nix: Home Manager's home.file.<name>.source needs the NixOS-level config, not the home-manager config. Pass it via extraSpecialArgs: home-manager.extraSpecialArgs = { inherit config; } in your flake, then accept it as an argument in home.nix.
Approach B — agenix-homemanager (community module) nix 
# There is a community project (agenix-homemanager / agenix-hm) that adds
# age.secrets directly inside Home Manager. It runs age as the user,
# not root. Simpler for user-owned secrets but requires a separate flake input.

# Flake input:
agenix-homemanager.url = "github:jordanisaacs/homemanager-agenix";

# home.nix:
imports = [ agenix-homemanager.homeManagerModules.default ];

age.secrets.brojo-ssh-key = {
  file = ../../secrets/brojo-ssh-key.age;
  path = "${config.home.homeDirectory}/.ssh/id_ed25519";
};

# Approach A (system-level) is more common and avoids the extra input.
✕
Do not set home.file.<name>.source to the /run/agenix/ path as a literal string. Use config.age.secrets.<name>.path so the reference is resolved at eval time, not at string interpolation time.
07

Adding and re-keying secrets

The agenix CLI is the only tool that touches .age files. It reads secrets/secrets.nix to know the recipient list, then opens your $EDITOR with the plaintext, and re-encrypts on save. You must be in the secrets/ directory (or pass --rules) so it finds the manifest.

Creating and editing secrets sh 
# Enter the dev shell so agenix is on PATH
nix develop

# Create a new secret (opens $EDITOR, saves encrypted .age file)
cd secrets
agenix -e brojo-ssh-key.age

# Edit an existing secret — must be able to decrypt it
# (requires your identity key: age key or the SSH key listed as a recipient)
agenix -e fuji-wireguard-key.age

# Re-key all secrets after adding a new host to secrets.nix
agenix -r

Re-keying is required whenever you add or remove a recipient from the manifest. The existing ciphertext was encrypted to the old recipient list; agenix -r decrypts every secret using your identity key and re-encrypts it to the updated list. You must have decrypt access to all secrets you re-key (i.e., your identity key must be in each secret's recipient list).

  1. Add new host key to manifest Edit secrets/secrets.nix to add the new host's SSH host public key to the relevant secret entries.
  2. Re-key Run agenix -r from the secrets/ directory. This re-encrypts every .age file to the updated recipient lists.
  3. Commit The .age files have changed (new ciphertext). Stage and commit them along with the updated secrets.nix.
  4. Deploy Run nixos-rebuild switch on each host. Activation decrypts the new ciphertext with the host's private key.
◆
Keep your author key (brojo in the manifest) in every secret. Without it you lose the ability to re-key from your local machine — you'd need to log into a host and extract the plaintext manually to recover.
08

The chicken-and-egg problem

The problem

To encrypt a secret for a new machine, you need its SSH host public key. But the host key doesn't exist until NixOS is installed — which requires the flake to evaluate cleanly — which requires the .age files to already exist.

There are three ways out of this loop:

  1. Install first, add secrets second (recommended) Install NixOS without the secret declared in the modules. After first boot, retrieve /etc/ssh/ssh_host_ed25519_key.pub, add it to the manifest, create/re-key the secrets, then re-deploy with the secret modules enabled. Suits the nixos-anywhere workflow well.
  2. Pre-generate the host key Generate an SSH host key pair locally before install. Commit the public key into the manifest, deploy the private key via some other mechanism (e.g., nixos-anywhere's --extra-files). The machine then has the expected host key from first boot and agenix activation succeeds immediately.
  3. Use a placeholder secret Create the .age file encrypted only to your author key (omit the host for now). Add the secret module to the flake. After install, get the host key, re-key, re-deploy. The placeholder means the flake evaluates without error; activation on the new host will fail until you've added the host to the manifest and re-keyed — but that's an acceptable transient state.
ℹ
After install, retrieve the host key with: ssh root@<ip> "cat /etc/ssh/ssh_host_ed25519_key.pub" Paste it into secrets/secrets.nix, run agenix -r, commit, then rebuild.
