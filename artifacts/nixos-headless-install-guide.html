Installing NixOS
USB-free, Fully Remote via nixos-anywhere

A guided walkthrough for provisioning a small headless server—running services like Gitea—without a USB stick, monitor, or keyboard. The entire install is driven from your local machine using nixos-anywhere, kexec, and disko.

01

Prerequisite knowledge

You don't need to be a sysadmin, but a handful of concepts are load-bearing for everything that follows. Understanding them now prevents confusion later.

Networking basics

Headless installation means your only interface is a network connection. You need to understand:

  • IP address — the numeric address your machine gets on your local network (e.g. 192.168.1.50). You'll use this to SSH in and to run nixos-anywhere.
  • DHCP vs static IP — DHCP means your router assigns the address automatically (it can change); a static IP is pinned and reliable for a server.
  • SSH (Secure Shell) — the protocol that gives you a remote terminal session. nixos-anywhere communicates entirely over SSH.
  • Ethernet required — nixos-anywhere does not support WiFi. The target machine must be on a wired connection during install.

New concepts for this method

  • kexec — a Linux kernel feature that loads and boots a new kernel from memory, replacing the running OS without a reboot cycle through the BIOS. nixos-anywhere uses this to swap the target's current OS for a NixOS installer environment, entirely over SSH, with no USB involved.
  • nixos-anywhere — a community tool that orchestrates the full install: it kexecs a NixOS installer onto the target, runs disko to format the disk, copies your flake config, runs nixos-install, then reboots. One command from your local machine.
  • disko — a NixOS module that declares disk partition layouts in Nix. nixos-anywhere requires disko because it needs to know how to partition the disk declaratively, without you running parted by hand. Your partition layout becomes part of your flake repo—reproducible and version-controlled.
The key shift in mental model

In the USB-based approach, you physically prepare the target (boot media, keyboard, parted). In this approach, the target is treated as a passive recipient—it just needs to be running any Linux and be reachable over SSH. Your local machine is the operator. Everything else is automated.

NixOS-specific concepts

  • Declarative configuration — you describe what the system should be; NixOS makes it so. Unlike Debian/Ubuntu where you imperatively install packages, NixOS state is fully described in .nix files.
  • nixos-rebuild switch — the command that reads your configuration and applies it to a running system. Used for all ongoing deployments after the initial install.
  • Flake outputs — a flake's outputs function is where nixosConfigurations lives. Each key in that attrset is a named system you can build and deploy to.

What you need on your local machine

sshClient for verifying initial connectivity to the target. Ships with macOS/Linux. nixYour local Nix installation. nixos-anywhere is run via nix run — no separate install needed. gitYour flake repo is tracked in git. Nix flakes refuse to include untracked files — everything must be committed.

What the target machine needs

any LinuxUbuntu, Debian, Arch — whatever came preinstalled. nixos-anywhere kexecs over it. SSH accessA user you can SSH into as root, or a user with passwordless sudo. Key auth preferred. ≥ 1 GB RAMkexec loads the NixOS installer entirely into memory. Less than 1 GB may cause failures unpacking the initrd. ethernetWiFi is not supported. Must be on a wired connection. x86_64The default kexec image targets x86_64-linux. aarch64 is possible with a custom kexec image — see the nixos-anywhere docs.
02

Mental models to adopt

These are the conceptual frames that make this approach feel logical rather than magical. Each one pays dividends across your entire Nix journey.

kexec is a kernel transplant

kexec doesn't reboot the machine in the traditional sense. It replaces the running kernel in-place from memory. nixos-anywhere uploads a minimal NixOS installer image and triggers kexec — the machine pivots from Ubuntu (or whatever) directly into NixOS installer space. Your SSH session drops and reconnects. That's normal.

The target's OS is disposable

Whatever is currently on the target machine — Ubuntu, Debian, a fresh-out-of-box OS — is irrelevant. nixos-anywhere obliterates it. The only thing that matters about the initial OS is that it's running, reachable over SSH, and supports kexec (essentially any x86_64 Linux).

Disk layout is code now

disko replaces the manual parted steps. You declare partition layout as a Nix module, commit it alongside your host config, and nixos-anywhere applies it. The disk layout is reproducible, version-controlled, and self-documenting — not a set of commands you'd have to remember or reconstruct.

Configuration before application

Write the full system config — SSH, users, services, disk layout — before running nixos-anywhere. The entire system is stamped in one shot. Your server boots fully configured and ready, not in a half-done state waiting for you to run more commands.

The deployment lifecycle

Confirm SSH access to target → write full config (disko + host + users) → commit to GitHub → run one nixos-anywhere command from local → machine reboots as NixOS → all future changes are nixos-rebuild --target-host from local. You never touch the target machine physically.

Ongoing deployment: pushing from local

This guide uses the push model throughout — your local machine is always the operator. After the initial install, config changes are applied with a single command:

initial install
nix run nixos-anywhere
ongoing changes
nixos-rebuild --target-host
builds on
Local machine, pushed to target
builds on
Local machine, pushed to target
target needs
Any Linux + SSH
target needs
Running NixOS + SSH + sudo

The target never needs git, nix, or your repo cloned onto it. Your local machine holds the full picture and pushes changes out.

03

Preparation

There is no USB stick to prepare. Preparation here means confirming your target is reachable and your local environment is ready to push to it.

1. Confirm SSH access to the target

Boot the target machine with whatever OS it has, make sure it's on ethernet, and confirm you can SSH in. This is your only physical interaction with the machine.

local machine
# Find the target's IP from your router's DHCP table, or run on the target itself:
#   ip addr show

ssh root@<target-ip>
# or a sudo-capable user:
ssh youruser@<target-ip>
i nixos-anywhere needs to either log in as root directly, or as a user with passwordless sudo. If the target's existing OS has a sudo user but requires a password, either temporarily grant passwordless sudo (NOPASSWD: ALL in sudoers) or use the --env-password flag with nixos-anywhere.

2. Copy your SSH public key to the target

nixos-anywhere uses your SSH key to authenticate. Make sure it's installed on the target now, so there are no password prompts during the automated install:

local machine
ssh-copy-id root@<target-ip>
# or for a non-root user:
ssh-copy-id youruser@<target-ip>

3. Find the target disk name

You'll need this for your disko config. SSH in and run:

remote shell
lsblk -d -o NAME,SIZE,TYPE,MODEL
# Look for your main disk — likely /dev/sda, /dev/nvme0n1, etc.
# Note the device path. You'll put it in disk-config.nix.
▲ nixos-anywhere + disko will completely wipe the target disk. Confirm you have the right device path before proceeding. There is no undo.

4. Verify kexec support

Almost all x86_64 Linux machines support kexec, but it takes two seconds to confirm:

remote shell
grep CONFIG_KEXEC /boot/config-$(uname -r)
# Should print: CONFIG_KEXEC=y
# If the file doesn't exist, try: zcat /proc/config.gz | grep CONFIG_KEXEC
04

Flake, disko & host structure

Before running nixos-anywhere, write the full configuration. This is the NixOS way: declare intent before applying it. We'll add a new host to your existing flake repo — let's call the machine gitara.

Repository layout

The key addition compared to a standard host is disk-config.nix — the disko partition layout that lives beside the hardware config:

directory layout
lib/
├── hardware/machines/
│   └── gitara/
│       ├── default.nix
│       ├── disk-config.nix          # NEW: disko partition layout
│       └── hardware-configuration.nix  # stub now; generated by nixos-anywhere at install
└── hosts/
    ├── default.nix                  # your existing productivity host
    └── server.nix                   # NEW: server host type

Add disko to your flake inputs

flake.nix · updated inputs
inputs = {
  nixpkgs.url = "github:nixos/nixpkgs?ref=nixos-unstable";
  nixos-hardware.url = "github:nixos/nixos-hardware";
  home-manager = {
    url = "github:nix-community/home-manager";
    inputs.nixpkgs.follows = "nixpkgs";
  };
  disko = {
    url = "github:nix-community/disko";
    inputs.nixpkgs.follows = "nixpkgs";
  };
};

The disk config (disko)

Create lib/hardware/machines/gitara/disk-config.nix. This replaces all manual parted and mkfs commands. Replace /dev/sda with the device path you noted during preparation:

lib/hardware/machines/gitara/disk-config.nix
{ ... }:
{
  disko.devices.disk.main = {
    device = "/dev/sda";  # adjust: /dev/nvme0n1 for NVMe
    type = "disk";
    content = {
      type = "gpt";
      partitions = {
        boot = {
          size = "512M";
          type = "EF00";  # EFI system partition
          content = {
            type = "filesystem";
            format = "vfat";
            mountpoint = "/boot";
          };
        };
        root = {
          size = "100%";
          content = {
            type = "filesystem";
            format = "ext4";
            mountpoint = "/";
          };
        };
      };
    };
  };
}

The hardware machine config

Create lib/hardware/machines/gitara/default.nix. It imports the disko module, the disk config, and a stub hardware-configuration that nixos-anywhere will replace:

lib/hardware/machines/gitara/default.nix
{ config, lib, pkgs, disko, ... }:
{
  imports = [
    disko.nixosModules.disko
    ./disk-config.nix
    ./hardware-configuration.nix
  ];
  hardware.enableAllFirmware = true;
}
▲ The hardware-configuration.nix import will fail evaluation if the file doesn't exist. Create a stub before committing: a file containing only { ... }: {}. nixos-anywhere overwrites it during install with the --generate-hardware-config flag.

The server host type

lib/hosts/server.nix
{ hostname ? "server" }:
{ config, lib, pkgs, ... }:
{
  imports = [ ../system/base-server.nix ];
  networking.hostName = hostname;
  networking.firewall.allowedTCPPorts = [ 22 3000 ];

  services.gitea = {
    enable = true;
    settings.server = {
      HTTP_PORT = 3000;
      DOMAIN = "gitara.local";
      ROOT_URL = "http://gitara.local:3000/";
    };
  };

  users.users.deploy = {
    isNormalUser = true;
    extraGroups = [ "wheel" ];
    openssh.authorizedKeys.keys = [
      "ssh-ed25519 AAAAC3... your-public-key-here"
    ];
  };

  # Required: nixos-rebuild --target-host needs passwordless sudo
  security.sudo.wheelNeedsPassword = false;
}

A minimal server base config

lib/system/base-server.nix
{ config, lib, pkgs, ... }:
{
  nix.settings.experimental-features = [ "nix-command" "flakes" ];
  nix.settings.auto-optimise-store = true;
  boot.loader.systemd-boot.enable = true;
  boot.loader.efi.canTouchEfiVariables = true;
  networking.networkmanager.enable = true;
  time.timeZone = "America/New_York";
  i18n.defaultLocale = "en_US.UTF-8";

  # SSH — the only way in. Do not omit.
  services.openssh = {
    enable = true;
    settings.PasswordAuthentication = false;
    settings.PermitRootLogin = "no";
  };

  nixpkgs.config.allowUnfree = true;
  environment.systemPackages = with pkgs; [ git vim curl wget ];
  system.stateVersion = "24.05";
}

Wire everything into flake.nix

disko must be passed via specialArgs so the hardware module can receive it as a function argument:

flake.nix · gitara system
outputs = { self, nixpkgs, home-manager, disko, ... }@inputs:
{
  nixosConfigurations = {
    jaro = ...;  # unchanged

    gitara = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      specialArgs = {
        inherit (inputs) nixos-hardware disko;
      };
      modules = [
        ./lib/hardware/machines/gitara
        (import ./lib/hosts/server.nix { hostname = "gitara"; })
      ];
    };
  };
};
→ Commit everything at this point — including the stub hardware-configuration.nix. Push to GitHub. nixos-anywhere can reference your flake directly from GitHub (github:youruser/system-conf#gitara), so the target never needs the repo on disk.
05

Execution

With your config committed and SSH access confirmed, the entire install is a single command from your local machine. nixos-anywhere handles kexec, disk formatting, installation, and reboot automatically.

1

Validate your config builds cleanly

Before touching the target, make sure your flake evaluates:

local
nix flake check
nixos-rebuild build --flake .#gitara
# Catches any evaluation or build errors before the point of no return.
2

Run nixos-anywhere

This single command kexecs a NixOS installer onto the target, runs disko to partition the disk, installs your configuration, generates hardware-configuration.nix locally, and reboots the machine:

local · the install command
nix run github:nix-community/nixos-anywhere -- \
  --flake .#gitara \
  --generate-hardware-config nixos-generate-config \
    ./lib/hardware/machines/gitara/hardware-configuration.nix \
  root@<target-ip>

What happens in sequence: nixos-anywhere SSHes in → uploads a kexec tarball → triggers kexec (the target's kernel is replaced in-memory with a NixOS installer; your SSH session will drop for ~30 seconds) → reconnects → runs disko to partition and format the disk → runs nixos-install with your flake → generates hardware-configuration.nix and writes it to your local repo → reboots the machine into NixOS.

i The SSH session drop during kexec is expected — not a failure. nixos-anywhere waits and reconnects automatically. The whole process typically takes 5–15 minutes depending on download speed.
3

Commit the generated hardware config

nixos-anywhere wrote hardware-configuration.nix into your local repo. Commit it so your flake is fully self-contained and repeatable:

local
git add lib/hardware/machines/gitara/hardware-configuration.nix
git commit -m "feat: add gitara hardware configuration"
git push
4

Verify the machine booted into NixOS

local
ssh deploy@<target-ip>
# Should connect with your key — no password prompt

uname -a
# Should show NixOS kernel

systemctl status gitea sshd

SSH running, Gitea running, your deploy user accessible. The machine is live.

Ongoing deployment workflow

All future config changes are pushed from your local machine. The target never needs git or your repo:

local · all future deploys
# 1. Edit config locally
vim lib/hosts/server.nix

# 2. Commit and push to GitHub
git add -A && git commit -m "feat: enable gitea backup"
git push

# 3. Build locally, push and activate on target
nixos-rebuild switch \
  --flake .#gitara \
  --target-host deploy@<target-ip> \
  --use-remote-sudo
→ nixos-rebuild builds the closure on your local machine (using your faster hardware and local Nix store cache), then copies only the changed derivations to the target and activates them. The target doesn't need to build anything itself.
06

Testing & verification

Testing in NixOS has two distinct phases: validating your config before deployment, and verifying the running system afterward. nixos-anywhere also has a built-in VM test mode that's worth knowing about.

Before deploying — validate the config

A

Evaluate the flake without building

local
nix flake check
# Evaluates all outputs for type errors and import errors.
# Catches typos and missing module references before touching the server.
B

Test disko layout in a VM (nixos-anywhere built-in)

nixos-anywhere can run your config in a throwaway VM and apply the disko partition script, verifying the disk layout is valid before touching real hardware:

local
nix run github:nix-community/nixos-anywhere -- \
  --flake .#gitara \
  --vm-test
# Boots a VM, runs disko, and installs your config inside it.
# No hardware touched. Surfaces disko errors and config build failures.
C

Build the closure locally

local
nixos-rebuild build --flake .#gitara
# Builds the full system closure and symlinks it to ./result.
# Confirms the config builds successfully. No side effects.
D

Dry activation (on the running server, for future changes)

local · after initial install
nixos-rebuild dry-activate \
  --flake .#gitara \
  --target-host deploy@<target-ip> \
  --use-remote-sudo
# Shows which systemd units would start/stop/restart
# without applying the change. Useful before any significant update.

After deploying — verify the running system

  • SSH connects as the deploy user with your key
  • systemctl status gitea shows active (running)
  • curl http://localhost:3000 returns Gitea's HTML
  • systemctl status sshd confirms SSH daemon is active
  • journalctl -u gitea -n 50 shows clean startup logs with no errors
remote · quick health check
# Service status
systemctl status gitea sshd networkmanager

# Check Gitea is reachable internally
curl -s http://localhost:3000 | grep -o "<title>.*</title>"

# Check which generation you're on
nixos-rebuild list-generations

# View recent system logs
journalctl -b --since "10 minutes ago"

Rollback — when things go wrong

NixOS's killer feature for server management: every nixos-rebuild switch creates a new generation. Rolling back is instantaneous:

remote · rollback
# Roll back to the previous generation
sudo nixos-rebuild switch --rollback

# Or choose a specific generation
sudo /run/current-system/bin/switch-to-configuration switch
nixos-rebuild list-generations
sudo nix-env --switch-generation <N> -p /nix/var/nix/profiles/system
The deeper takeaway

Every generation is a complete, immutable snapshot of your system. Unlike traditional servers where you might spend hours debugging a bad upgrade, NixOS rollbacks are atomic—the system either fully reflects a generation, or it doesn't. For a self-hosted server, this is an extraordinary safety net.

→ Access Gitea from your local network by navigating to http://<server-ip>:3000 in a browser. On first visit, Gitea presents a setup wizard where you'll create the admin account. Configure your ROOT_URL in the Nix config to match however you're accessing it.
