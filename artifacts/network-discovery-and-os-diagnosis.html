Network Discovery
& OS Diagnosis of an Unknown Host

Before you can nixos-anywhere a machine, you must locate it on the LAN, figure out what (if anything) is running on it, and decide how it will hold a stable address. This guide walks the diagnostic phase that precedes the install — a learning track as much as a runbook.

Prerequisite vocabulary

The diagnostic phase trades almost entirely in three layers of identity. If you internalise the distinction between them, every tool in this guide stops feeling arbitrary and starts feeling like a way of poking one specific layer.

Layer
Identifier
Lifetime & mutability
link (L2)
MAC address — a4:bb:6d:..
Burned into the NIC. Effectively permanent. The only stable identity until you have one.
network (L3)
IP address — 192.168.1.50
A lease, not a property. Can change at any DHCP renewal unless pinned.
application
hostname — gitara.local
A label resolved by DNS / mDNS. Convenient, but useless until DNS knows about it.

Pre-install, the IP and hostname don't exist yet in any reliable sense. The MAC address is the one identifier that survives a reboot, an OS swap, and a router restart — so it becomes the anchor point for everything you do.

Core terms

ARPAddress Resolution Protocol — the LAN's way of asking "who has IP 192.168.1.50?" and getting back a MAC. Your kernel maintains an ARP cache; ip neigh dumps it. DHCPDynamic Host Configuration Protocol — the negotiation where a fresh host asks the LAN "give me an IP" and a server (your router) replies with a lease. DHCP leaseA time-bound assignment: IP X belongs to MAC Y until time Z. Renewed periodically. Can be revoked or changed. DHCP reservationA router-side rule: "whenever MAC Y appears, always give it IP X." Same protocol, but the answer is pinned. Often called "static DHCP". static IPAn address configured on the host itself, bypassing DHCP entirely. Brittle if the LAN's subnet changes; the router has no idea. subnet / CIDRThe range of addresses on your LAN, written like 192.168.1.0/24. The /24 means "first 24 bits are the network, last 8 are host" — i.e. 256 possible addresses. mDNS / .localMulticast DNS — a way for hosts to broadcast their own name (foo.local) without a DNS server. Linux via avahi, macOS native, Windows partial. TTLTime-to-live on an IP packet. Different OS families ship different defaults — a useful fingerprinting signal.

What you need on your local machine

nmapSwiss-army-knife scanner. Does ARP sweeps, port scans, OS fingerprinting, banner grabs. arp-scanFaster and louder than nmap for pure L2 discovery. Bypasses ICMP entirely. ip / iproute2Inspect your own interfaces, routing table, and ARP cache. ip a, ip r, ip neigh. sshFor the moment you've found the host and want to confirm it speaks SSH. router admin UIBrowser access to your gateway (often 192.168.1.1). The DHCP client list is the single most useful page you'll touch.
02

Mental models

Four frames that make the rest of this guide click. They're worth re-reading once after you've finished — the diagnostic flow makes more sense from the other side.

The MAC is the anchor

An unknown box on your LAN has no stable IP, no DNS name, no SSH key you know. The MAC of its NIC is the one identifier you can latch onto and follow across reboots, reinstalls, and IP changes. Every "find it again later" mechanism — DHCP reservation, ARP lookup, router tables — is keyed on MAC.

Discovery is a layered question

"Is the machine alive?" is L2 (ARP). "Does it route?" is L3 (ping). "Does it speak SSH?" is L4 + application. Each layer can succeed while the one above it fails — a host can ARP-respond while ignoring pings, or accept TCP while rejecting your key. Diagnose top-down or bottom-up, but know which layer you're testing.

OS fingerprinting is inference

You never directly observe what OS is running. You observe behaviour — open ports, packet TTLs, TCP/IP stack quirks, service banners — and infer. Each signal is a probability shift, not proof. Two strong independent signals (e.g. TTL 128 + port 3389 open) are more trustworthy than one strong signal.

Addressing is a negotiation

"Static IP" feels like a property of the machine, but the LAN doesn't care what the machine claims if the router hands the same address to someone else. A DHCP reservation on the router is almost always the right answer for a server — the address is enforced by the authority that issues addresses, not by the recipient.

The diagnostic arc

Start from nothing → enumerate every MAC on the LAN → narrow to the candidate by elimination or by hardware MAC OUI → confirm it's the right box → fingerprint the OS to know what bridge you need to cross → pin the address by MAC so it stays found → cross the bridge to a state where nixos-anywhere can take over. Each phase reduces uncertainty before the next one begins.

A

Phase A · Network reconnaissance

The goal of this phase is a list of every host on your subnet with their MAC and IP, from which you'll identify the target by elimination. You aren't trying to talk to the target yet — you're trying to locate it.

A1 · Know your own subnet

Before you can sweep the LAN, you must know which addresses count as "the LAN".

local · identify your subnet
ip -4 addr show
# Look for your active interface (often wlan0 / enp*s* / eth0).
# You'll see something like:
#   inet 192.168.1.42/24 brd 192.168.1.255 scope global ...

ip route show default
# Confirms your gateway address — usually the router itself.
# default via 192.168.1.1 dev wlan0

From that you know two things: the subnet to sweep (192.168.1.0/24 in the example) and the router (192.168.1.1), which is your second source of truth in the next step.

A2 · The DHCP client list (the easy answer)

If the target machine got a DHCP lease — which it will if it boots into any modern OS with default networking — your router already knows about it. Most consumer routers expose this under "Connected devices", "DHCP client list", or similar.

→ This is almost always the fastest path. Plug the new machine in, power it on, wait ~60 seconds, refresh the router's client list, look for the new entry. The MAC and hostname (if the OS volunteered one) appear there.

A3 · ARP sweep (the authoritative answer)

If the router UI is unhelpful, or the host hasn't completed DHCP, an ARP scan asks the LAN directly. ARP works below ICMP — even hosts that drop pings will respond to ARP, because the local link layer requires it.

local · ARP-level discovery
# Fastest, most reliable for L2-reachable hosts. Requires root.
sudo arp-scan --localnet
# Or, if not installed:
sudo nmap -sn -PR 192.168.1.0/24
# -sn  = no port scan, just host discovery
# -PR  = ARP discovery (default on a local subnet anyway)

Output is a list of IP/MAC pairs and, often, a vendor name decoded from the MAC's OUI (the first 24 bits identify the NIC manufacturer). A line like 192.168.1.73 a4:bb:6d:11:22:33 Intel Corporate is enough to recognise a new desktop with an Intel NIC sitting next to a sea of Apple and Samsung phones.

A4 · Identify the target by elimination

Two strategies, depending on what you know:

  • You know the target's MAC (BIOS shows it, sticker on the NIC, manufacturer plate). Grep the scan output for those octets. Done.
  • You don't know the MAC. Run the scan once before plugging in the new machine, save the output. Power on the new machine, wait, run the scan again. The new line is the target.
local · before / after diff
sudo arp-scan --localnet > /tmp/lan-before.txt
# ... plug in and boot the target ...
sudo arp-scan --localnet > /tmp/lan-after.txt
diff /tmp/lan-before.txt /tmp/lan-after.txt
i If the target shows up with no IP, only a MAC in the router's tables — it powered on, sent a DHCP request, but didn't complete. Likely a firewall, a misconfigured NIC, or a non-networked installer waiting for input. Check the cable and the link light first.

A5 · Sanity-check the candidate IP

local · liveness ping
ping -c 4 192.168.1.73
# Some hosts (Windows by default) drop ICMP. If ping fails but the
# host showed up in ARP, the host is alive — just silent at L3.

ip neigh show
# Shows your local ARP cache. The candidate should appear with
# state REACHABLE or STALE — both mean the kernel has confirmed it.
B

Phase B · OS fingerprinting

You now have an IP. The next question is what's behind it. Three signals, used together, will tell you with high confidence whether you're looking at Linux, Windows, a vendor installer, or nothing at all.

Signal 1 · The port surface

Different OS families ship different default services. The port profile is often the loudest signal.

22SSH. Strong signal for Linux / BSD / macOS. (Windows can run OpenSSH but rarely does by default.) 135 / 139 / 445SMB / NetBIOS. Strong signal for Windows. Samba on Linux also opens these, but the combination of all three is Windows-shaped. 3389RDP. Effectively Windows-only by default. 5357WSDAPI — Windows network discovery. Windows-only. 631CUPS printing. Common on Linux / macOS. 5353mDNS / Bonjour / avahi. Common on macOS and Linux desktops; less common on Windows. nothing openEither a strict firewall, a freshly-booted installer with no services, or no OS at all (PXE/BIOS waiting).
local · scan the candidate
nmap -Pn -sS -p- 192.168.1.73
# -Pn  = assume host is up; skip ICMP discovery (works behind firewalls)
# -sS  = TCP SYN scan (fast, requires root)
# -p-  = all 65535 ports. Slower; use a top-1000 scan first if needed:

nmap -Pn 192.168.1.73
# Top 1000 ports is usually enough to fingerprint the OS family.

Signal 2 · TTL — the cheapest possible signal

The TTL field in an IP packet is decremented at each router hop. On a local LAN you see roughly the OS's initial TTL. The defaults vary by OS family:

TTL ~64Linux, BSD, macOS, Android. Most modern Unix-derived stacks. TTL ~128Windows. TTL ~255Cisco / routers / some embedded.
local · read TTL off a ping
ping -c 1 192.168.1.73
# 64 bytes from 192.168.1.73: icmp_seq=1 ttl=64 time=0.4 ms
#                                              ^^^^^^
# Initial TTL was 64 → almost certainly Linux/BSD/macOS.
# If you see ttl=128 → Windows. ttl=127 → Windows one hop away. Etc.
▲ TTL is a hint, not proof. It's trivially forgeable, can be altered by sysctl, and a router between you and the host shifts it. Treat it as a vote, not a verdict.

Signal 3 · Active fingerprinting (nmap -O)

nmap's -O probes TCP/IP stack idiosyncrasies that differ subtly across OS families — initial sequence number patterns, window sizes, TCP options ordering — and matches them against a fingerprint database.

local · active OS detection
sudo nmap -O 192.168.1.73
# Requires at least one open and one closed port to be reliable.
# Outputs a guess like:
#   Running: Linux 5.X
#   OS CPE: cpe:/o:linux:linux_kernel:5
#   OS details: Linux 5.4 - 5.15

# Add service / banner grabbing for more confidence:
sudo nmap -A 192.168.1.73
# -A folds in -O, -sV (service versions), default scripts, traceroute.
# SSH banners often leak the distro (Ubuntu 22.04, Debian 12, etc.).

Putting the signals together

Profile · Linux

Port 22 open + TTL 64 + SSH banner mentions OpenSSH / Ubuntu / Debian. Reachable. You're ready for nixos-anywhere.

Profile · Windows

Ports 135/139/445 open, often 3389 + 5357. TTL 128. SMB banner identifies Windows version. Needs a Linux bridge first.

Profile · No OS / PXE

Host appears in ARP, but no ports open and no ICMP. Often a UEFI shell, a BIOS PXE-boot wait, or a halted installer. Needs hands-on or PXE boot.

Profile · Linux installer / live USB

Port 22 open if the live OS started sshd, often with a default password. Limited disk. Useful as a transient bridge to install NixOS. Confirm before assuming.

→

Decision point · branch by state

The diagnosis collapses into one of four states. Each branches to a different path through the rest of this guide.

┌─ Linux + SSH reachable → Phase C → install guide. Easiest path. ├─ Linux but SSH not exposed → Enable sshd locally (one-time hands-on), then Phase C. ├─ Windows of any flavour → Phase D (Linux bridge) → Phase C → install. └─ No OS / unresponsive → Phase D (boot live env, PXE, or USB), then Phase C.
Why this branching matters

The install tool only knows one trick: it SSHes in, kexecs over a running Linux. Every branch that isn't already "Linux + SSH" is just a different way to get there. Once you reach the green state, the existing install guide takes over and the box is on track to be NixOS.

C

Phase C · Pin a stable address

Whatever IP the box currently holds is a lease — it can change on the next renewal. Before you invest in this address (SSH config, future nixos-rebuild --target-host, DNS entries), pin it. There are three places this can happen, and they're not equivalent.

option 1 · best
DHCP reservation on the router
option 2 · ok
Static IP on the host
option 3 · soft
mDNS hostname (.local)

Option 1 · DHCP reservation on the router (recommended)

The router binds a specific MAC to a specific IP. The host continues to do DHCP — it just always gets the same answer. The authority that owns address allocation is the one enforcing the pin, which is what makes this the cleanest model.

  • Log into the router's admin UI (usually http://<gateway-ip>).
  • Find LAN → DHCP server → Address reservation (names vary by vendor).
  • Add a rule: MAC aa:bb:cc:dd:ee:ff → IP 192.168.1.50.
  • Choose an IP outside the router's dynamic pool if the UI offers that distinction, to avoid future collisions.
  • Reboot the target (or run dhclient -r && dhclient) so it picks up the new lease.

Option 2 · Static IP configured on the host

The host claims an address without asking. Works, but the router has no idea — if a different DHCP client gets handed the same address, you'll have a conflict the static host will lose on the next ARP storm.

▲ If you go this route, the chosen IP must be outside the router's DHCP pool. Most home routers default to a pool like 192.168.1.100–199; picking 192.168.1.50 (below the pool) is safe.

Option 3 · mDNS hostname (complement, not replacement)

If the host runs avahi (most Linux distros) or Bonjour (macOS), it advertises itself as <hostname>.local on the LAN. You can SSH to gitara.local without knowing or caring what IP it currently holds.

local · try mDNS
avahi-resolve -n4 gitara.local
# Or just:
ping gitara.local

Treat this as a convenience layer on top of one of the pinning options, not a substitute for them. It depends on avahi being running and on every client supporting mDNS.

D

Phase D · Bridge to install-ready

This phase only applies to the non-green branches: Windows installations, no OS, or Linux with SSH disabled. The endpoint is the same in every case — a running Linux with sshd listening on port 22 and your public key authorised.

Branch · Windows on disk

You cannot kexec from Windows. The cleanest path is to boot a Linux live environment instead, which uses RAM and ignores the Windows install on disk entirely. Once nixos-anywhere runs, that Windows install gets wiped along with everything else.

  • Path 1 — Linux live USB (least friction): write a NixOS minimal ISO or Ubuntu live to a USB stick, plug it in, boot from it once. Enable sshd, set a password or drop your key, then run the install from your local machine.
  • Path 2 — PXE / netboot.xyz: if the BIOS supports network boot, you can chainload a live environment over the LAN with no USB. Setup is heavier — you need a TFTP/HTTP server on the LAN — but no physical media touches the target.
  • Path 3 — IPMI / BMC: if the box has out-of-band management (server hardware does, most desktops don't), you can mount an ISO over the network and boot it remotely.
i "Headless remote install" stops being literally true the moment you discover the target is Windows. At minimum you need a USB stick and one boot-time keypress. After that one moment, the rest of the install is fully remote.

Branch · No OS / fresh hardware

Same paths as the Windows case — the disk is empty so there's nothing to preserve, but you still need to put something bootable in front of the BIOS. A NixOS minimal ISO on USB is the most direct option, and it includes sshd ready to enable.

on the live-booted target · enable ssh
# NixOS minimal ISO already has openssh available. Start it:
sudo systemctl start sshd

# Set a password for root (no password by default on the ISO):
sudo passwd root

# Confirm the IP — needed for the install command:
ip -4 addr show

# Optional: drop your public key in directly to skip the password prompt
mkdir -p /root/.ssh
curl https://github.com/<you>.keys > /root/.ssh/authorized_keys

Branch · Linux with SSH disabled

The host runs Linux but isn't exposing port 22. If you have physical or KVM access, the fix is a one-liner on the host:

on the target · enable + open ssh
sudo systemctl enable --now sshd
sudo ufw allow 22       # Ubuntu / Debian if ufw is in use
sudo firewall-cmd --add-service=ssh --permanent  # Fedora / RHEL

The handoff

You're done with this guide when all of the following are true. The install guide picks up from here.

  • You know the target's MAC address.
  • The target has an IP that won't change (DHCP reservation or static).
  • ssh root@<ip> or ssh <user>@<ip> with passwordless sudo lands you in a shell.
  • The shell is running Linux (any distro, including a live ISO).
  • The target is on ethernet, not WiFi.
→ Once those boxes are checked, the existing USB-free, Fully Remote via nixos-anywhere guide takes over. The diagnostic uncertainty is gone — from here on, every step is deterministic.
